## 未決定事項（TBD）一覧

実装前に決めると、後工程の手戻りが大きく減る項目を列挙します。

- 追加提案は `11_questions_and_proposals.md` の「2.6 未決定仕様への追加提案パッケージ（要件定義前進用）」を参照。

### ルール/フォーマット
- （確定）サポートカードは **クールダウン制**（複数回使用可。`cooldownTurns` で再使用制限）→ `05_burst_and_support.md`, `07_data_model.md`
- （確定）**制限カード/禁止カード制度**を導入する。制限カードは「チームのデッキに1枚まで」のような制約を課す（`01_core_rules.md`）
- （確定）対戦ルールは **`RegulationDefinition`** でパラメータ化。色制約・同名制限・最大ターン数等を柔軟に定義可能 → `07_data_model.md`
- （確定）**引き分け回避の勝敗判定**：「ダウンしていないキャラのHP割合合計（hpRatioSum）」で判定。完全同率時の最終タイブレークは `initiativeTeamId` → `01_core_rules.md`
- （確定）**最大ターン数**：`RegulationDefinition.maxTurns` で定義（標準値：15）。到達時はhpRatioSumで判定 → `01_core_rules.md`

### リソース（コスト）
- （確定）**AP消費の会計モデル**：APは予約時に仮押さえしない。バトルフェイズの実行時に消費。不発時はAP消費なし・手札に残る・相手に非公開 → `06_turn_flow.md`
- （確定）**コスト踏み倒し下限**：`max(1, originalCost - reduction)`（最低1AP消費を保証）
- コスト踏み倒しの許容範囲（特に橙/サポート）は個別カード設計で調整

### 盤面
- （確定）**召喚枠の上書きルール**：上書き時は `SummonReplaced` イベント発火。on-expireトリガーは上書き時に発火しない。同一ターン複数召喚は通常解決順で処理 → `03_summons.md` セクション6

### 解決順/対抗手段
- （確定）**同速タイブレークの完全チェーン**：`priority降順 > spd降順 > decisionSubmittedAtMs昇順 > initiativeTeamId > createdSeq昇順`。乱数による最終タイブレークは行わない → `06_turn_flow.md`
- （確定）**イニシアチブ**：ゲーム開始時に `rngSeed` から決定し、**以降固定**（更新なし）
- （確定）準備フェイズ中の予約上限：各キャラの **現在APまで** 予約可能（複数予約可、AP消費はバトルフェイズで実行時）→ `06_turn_flow.md`
- 同時誘発の解決順（所有者選択 vs 固定優先順位、両者同時の処理）
- サポートに対する対抗手段（無効化/軽減/封印など）を入れるか
- （確定）**トリガーシステム**：カードテキストでトリガー条件を定義し、バトルフェイズ中に割り込み早期発動が可能。未達時は通常通りpriority順で発動 → `02_cards.md` セクション7
- （確定）**継続効果の合成順**：ステータス修正パイプライン（base → additive → multiplicative → clamp → finalFloor）で統一。同カテゴリ内は `createdSeq` 昇順 → `07_data_model.md` セクション5

### バースト（ゲージ/バースト状態）
- BurstMax=5 は確定。増加条件として「決定速度ボーナス」「行動ごと（+30pt）」を採用済み。
- （確定）バースト状態のステータス上昇：**倍率式**（pAtk/mAtk ×1.15、pDef/mDef/spd/tech ×1.10、maxHp変動なし）→ `07_data_model.md`
  - Burst倍率の端数は切り捨て（floor）確定

### 優先度（priority）
- （確定）標準レンジ：`+3, +2, +1, 0, -1, -2, -3`

### エレメント（Element）

### 召喚入れ替え・コスト軽減

### デッキ（切れなし）
- （確定）**`DeckReshuffledFromTrash` イベント仕様**：`turn`, `teamId`, `trashReturnedCount`, `banishedIgnoredCount`, `rngSeedSnapshot`, `rngStepBefore/After` を保存。シャッフルアルゴリズムは **Fisher-Yates**（推奨）→ `07_data_model.md`
- トークンカード（デッキ外生成）の識別方法と、使用後に `banished` へ送る仕様のログ表現（TBD）

### セット効果（構築時パッシブ）
- （確定）「ゲーム開始時追加ドロー」系セット効果は、**初期手札5枚ドローの後** に処理する
- （確定）サポートカードにもセット効果を持てる。スコープは原則 `team`（チーム全体に適用）

### 召喚の寿命・継承

### 色/デッキ制約
- （確定）色制約は `RegulationDefinition.colorRestriction` で定義（`none` / `maxColors` / `singleOnly`）
- （確定）同名制限は `RegulationDefinition.sameNameRule` で定義（`perCharacterDeck` / `perTeam`）。標準値：チーム全体で同名2枚まで
- （確定）無色カード制限は `RegulationDefinition.colorlessCardLimit` で定義（`none` / `ratio` / `fixed`）。標準値：デッキの半数まで

### ユニーク/パッシブ解放
- （確定）ユニークカードのクールダウン減少タイミング：**ターン終了時**（他の持続カウントと統一）
- キャラパッシブの解放条件（unlockedMaxAP閾値）の標準値と例外許容（例：4/6/7を全キャラ固定にするか）

### 状態異常/バフ（StatusEffect）
- （確定）同一状態の重ねがけ：`replace`（上書き）/ `extend`（延長）/ `stack`（重ね掛け）の3モード → `07_data_model.md`
- （確定）`stack` の `maxStacks` はカード/状態定義ごとに個別設定可能
- （確定）持続ターンの減少タイミング：**ターン終了時** に統一
- （確定）**解除（dispel）/無効化（immune）の共通ルール**：タグベース（`purgeable`, `crowdControl`, `dot`, `hot`, `statModifier`）で判定。dispelは解決時、immuneは付与時に判定 → `02_cards.md` セクション9

### ダメージ計算式
- （確定）**混合式**：`max(1, floor(ATK × カード倍率 × 100 / (100 + DEF × 防御係数)))` → `15_damage_formula.md`
- （確定）防御係数の初期値：**0.6**
- （確定）最低1ダメージ保証
- （確定）属性相性倍率は **なし**
- （確定）**tech計算式**：クリティカル率（`clamp((attackerTech - defenderTech×0.5)/500, 0.02, 0.30)`、倍率1.5固定）＋状態異常命中率（`clamp(baseRate + (attackerTech - defenderTech)×0.001, 0.10, 0.95)`）→ `15_damage_formula.md` セクション8

### 召喚ユニットのステータス再計算
- （確定）召喚ユニットのステータスは **召喚時に確定**（リアルタイム追従なし）

### アイテムカードのセットタイミング
- （確定）準備フェイズで予約→バトルフェイズでセット実行
- （確定）AP消費はセット時のみ。自動発動は0コスト
- （確定）セットと自動発動は同一ターン内に起こりうる

### 蘇生
- （確定）蘇生はカード効果として **制限なし** で許可。勝敗判定は **バトルフェイズの全行動解決後** に行う → `01_core_rules.md`, `06_turn_flow.md`

### 召喚のゾーン管理
- （確定）召喚カード使用→カードは **トラッシュ（graveyard）** へ。召喚実体が消滅→実体は **追放ゾーン（banished）** へ → `03_summons.md` セクション7

### 状態異常/バフのフレームワーク
- （確定）`StatusEffectDefinition` テンプレートを導入。新しい状態異常/バフを順次追加可能 → `07_data_model.md`

### 召喚オブジェクト→ユニットの解決順保証
- 同一ターンにオブジェクトとユニットの両方を予約した場合の順序保証（priority設計 or 1予約化）

### バースト適用と同ターン召喚の順序
- （確定）バトルフェイズ開始時にバースト状態適用→既存召喚のバースト変化。同ターンの召喚カードは通常の解決順で処理されバースト版として出現する → `06_turn_flow.md`

### 1vs1方式B固有
- サポートのユニーク装備の意義（使用不可なら装備を省略するか）
- HP割合タイブレークでサポートのHPを含むか

### 新メカニクス（EffectType追加分）
- **Discard（手札捨て）**: 相手の手札を捨てさせるカード（ハンデス）のバランス上限。1ターンに何枚まで捨てさせられるか
- **Bounce（バウンス）**: 召喚ユニットをバウンスした場合、その召喚実体はどうなるか（手札に戻る＝召喚カードが手札に戻る？ or 消滅扱い？）
- **DeckSearch**: サーチ対象が見つからなかった場合のデッキ情報漏洩（相手にデッキ内容が推測される問題）への対応
- **TokenCreate**: トークンカードの定義方法（テンプレートID参照 vs 動的生成）の最終決定
- **Reveal（手札公開）**: 公開された情報をUI上でどう表示するか（Phase 2以降）
- **可変コスト（X AP）**: 「残りAPすべて消費」のコスト表現をどう EffectSpec/PlannedAction に組み込むか

## 要件を深掘りしたい優先箇所（レビュー観点）

  - `priority降順 > spd降順 > decisionSubmittedAtMs昇順 > initiativeTeamId > createdSeq昇順` で確定 → `06_turn_flow.md`
- **（確定）AP消費の会計モデル**
  - 予約時に仮押さえしない。実行時消費。不発時はAP消費なし・手札に残る → `06_turn_flow.md`
- **（確定）継続効果の合成順**
  - base → additive → multiplicative → clamp → finalFloor のパイプラインで確定 → `07_data_model.md` セクション5

### 深掘り時の推奨アウトプット形式
- 各論点について、最低でも次の4点を定義する。
  - **ルール本文（1〜3行）**
  - **例外規則（不発/同値/上限超過時）**
  - **ログ項目（何をイベントに残すか）**
  - **テスト観点（正常系/境界値/同時発生）**
